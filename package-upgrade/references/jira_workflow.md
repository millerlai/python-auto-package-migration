# Jira Integration Workflow

當使用者用 Jira URL / Jira ID 觸發 skill (Phase 1 情況 C),完整流程在這裡。
SKILL.md 是高階指引,這份是實作細節 + 範本。

---

## 1. 輸入解析

### 1.1 支援的輸入格式

| 輸入 | 抽出結果 |
|------|----------|
| `https://trendmicro.atlassian.net/browse/V1E-148968` | site=`trendmicro.atlassian.net`, key=`V1E-148968` |
| `https://trendmicro.atlassian.net/jira/software/c/projects/V1E/issues/V1E-148968` | site=`trendmicro.atlassian.net`, key=`V1E-148968` |
| `V1E-148968` (純 key) | key=`V1E-148968`, site=查 memory 或詢問 |
| 任何含 `/browse/<KEY>` 的 URL | 從 hostname 抽 site, 從 path 抽 key |

### 1.2 Regex pattern

```python
# 完整 URL
URL_PATTERN = r'https?://([^/]+)/browse/([A-Z][A-Z0-9]+-\d+)'
# 純 issue key
KEY_PATTERN = r'\b([A-Z][A-Z0-9]+-\d+)\b'
```

### 1.3 Default site (auto-memory)

當第一次成功用某個 site 時,將其存為 reference type memory:

```yaml
name: atlassian_default_site
type: reference
description: User's default Atlassian site for Jira tickets when only an issue key is given
content: trendmicro.atlassian.net
```

下次只給 key 時直接使用此 default,不再詢問。

## 2. 抓取 ticket — MCP vs REST decision tree

```
是否有 Atlassian MCP (claude.ai connector)?
├── 是 → 嘗試 mcp__claude_ai_Atlassian_Rovo__getJiraIssue
│        ├── 200 OK → 用 MCP 路徑, auth_mode = "mcp"
│        └── 401/403/unauthorized → fallback ↓
│
└── 否 / fallback → 詢問使用者選 [1] [2] [3] (見 SKILL.md Phase 1.C.2)
         ├── [1] 提供 token → scripts/jira_fetch.py, auth_mode = "rest_token"
         ├── [2] 重試 MCP → 回到頂端 (使用者重新登入後)
         └── [3] 手動貼 → 使用者直接在對話中貼內容, auth_mode = "manual"
```

`auth_mode` 必須在 `jira_context` 中保留,Phase 7.5/7.6 用同一個方式。

### 2.1 MCP 呼叫範本

```
mcp__claude_ai_Atlassian_Rovo__getJiraIssue(
  cloudId="trendmicro.atlassian.net",
  issueIdOrKey="V1E-148968",
  responseContentFormat="markdown",
  fields=["summary", "description", "status", "labels", "comment", "issuetype", "priority"]
)
```

注意 MCP 的 `cloudId` 接受 site hostname,內部會自動解析到實際 cloud ID。
如果 MCP 回 "cloudId not recognized" 之類錯誤,改呼叫
`getAccessibleAtlassianResources` 取得實際 UUID,然後再呼叫 `getJiraIssue`。

### 2.2 REST fallback

```bash
ATLASSIAN_EMAIL=miller_lai@trendmicro.com \
ATLASSIAN_API_TOKEN=ATATT3xFfGF0... \
  python scripts/jira_fetch.py trendmicro.atlassian.net V1E-148968
```

Output 是 JSON,結構:
```json
{
  "key": "V1E-148968",
  "url": "https://trendmicro.atlassian.net/browse/V1E-148968",
  "summary": "Upgrade requests to fix CVE-2024-35195",
  "status": "In Progress",
  "issue_type": "Task",
  "priority": "High",
  "labels": ["security", "dependency"],
  "description": "...",
  "comments": [{"author": "...", "created": "...", "body": "..."}]
}
```

### 2.3 Token 取得指引

要求使用者前往: <https://id.atlassian.com/manage-profile/security/api-tokens>
建立 API token (不是密碼),範圍至少要能讀寫 Jira。

⚠️ **token 出現在這個對話的 transcript 中**。建議使用者:
- 用完之後到 Atlassian 把 token revoke
- 或選擇 [3] 手動貼,完全避免 token 進入對話

## 3. 從 ticket 抽取升級資訊 (LLM 任務)

### 3.1 套件名與版本

優先順序:
1. **Title / Summary**: 通常最明確,例 `Upgrade requests to 2.32.0`
2. **Description 程式碼區塊**: 找 ` ```...``` ` 內的 `requirements.txt`
   或 `pyproject.toml` 片段
3. **Description 內容**: 找以下 patterns:
   - `<package>==<version>` / `<package>>=<version>`
   - `bump <package> to <version>`
   - `upgrade <package> from X to Y`
   - `<package> X.Y has CVE` (推論 fix version 要查 advisory)
4. **Labels**: 例如 label `requests` 暗示套件名
5. **Linked CVE**: 若 ticket body 含 CVE-XXXX-XXXXX → 走情況 B 流程

### 3.2 驗收條件 (Acceptance Criteria)

找以下 section heading (case-insensitive):
- `Acceptance Criteria` / `AC` / `驗收條件`
- `Definition of Done` / `DoD`
- `Tasks` / `Checklist` (若是 checkbox 列表)

把每條條件留下來,Phase 7.5 的 comment 會逐條標記完成狀態。

### 3.3 不要過度推論

如果 ticket 描述模糊 (例 "升級安全相關的套件"),不要猜。在 Phase 1.C.4
直接告訴使用者「ticket 沒有明確指定哪個套件,請告知」。

## 4. Comment Template (Phase 7.5)

```markdown
## 🤖 Automated upgrade by Claude Code

**Package**: `{package}` `{old_version}` → `{new_version}`
**Branch**: `{branch_name}`
**PR**: {pr_url_or_"未建立 (僅本機 commit)"}
**Commit**: `{commit_sha_short}`
**Tests**: ✅ Passed ({test_count} tests) | ❌ {failed_count} failures

### Executive Summary
{Phase 7.1 第一節原文 — 3-5 句話}

### Breaking Changes
- 處理 {N} 個 breaking changes
- 修改 {M} 個檔案、{K} 處變更
- 詳情見 PR description

### Acceptance Criteria
{逐條標記}
- ✅ {criterion 1}
- ✅ {criterion 2}
- ⏭️ {criterion that requires manual verification — 不能自動勾,留給 reviewer}

### Rollback
若需要回退:
```bash
git checkout {original_branch}
git branch -D {feature_branch}
bash scripts/snapshot_env.sh {project_path} restore
```

---
*Generated by [package-upgrade skill](https://github.com/.../package-upgrade-skill) at {timestamp}*
```

### 4.1 Redaction 注意事項

⚠️ comment 對 ticket watchers 都看得到。Post 之前檢查內容是否含:

- ❌ Token / 密鑰 (即使是被 revoke 的舊 token 也不要 post)
- ❌ Internal-only 路徑 (如 `/Users/<personal>/...`)
- ❌ 客戶資料 / PII (若 test fixture 中有)
- ✅ 套件版本、檔案路徑 (相對於 repo root)、commit SHA、PR URL

如果報告中有 sensitive 內容,在 7.5.2 的預覽階段告訴使用者並建議刪除。

## 5. Status Transitions (Phase 7.6)

### 5.1 Done 同義詞 match 順序

```python
DONE_SYNONYMS = ["done", "resolved", "closed", "completed", "fixed"]
```

從 `getTransitionsForJiraIssue` 回傳的清單中,case-insensitive 比對 transition
**name** 和 **to_status**。第一個 match 即為 default 推薦。

範例 — Trend Micro V1E project 的常見 transitions:
- `In Progress` → `Code Review` (id 31)
- `Code Review` → `Done` (id 41) ← match
- `Done` → `Reopen` (id 51)

若清單中有多個 match (例如同時有 `Done` 和 `Resolved`),取第一個並在
prompt 中列出其他選項讓使用者改選。

### 5.2 Resolution field

某些 workflow 在 transition 到 Done 時要求 `resolution` 欄位。
預設值規則:

| 觸發來源 | 預設 resolution |
|----------|-----------------|
| CVE 修復 (Phase 1 情況 B 或 ticket 含 CVE) | `Fixed` |
| 一般 package upgrade | `Done` |
| 不確定 / workflow 不需要 | 不設定 |

若 transition 因 resolution 欄位缺失被拒 (HTTP 400 with field validation),
從錯誤訊息抽出可用的 resolution 選項,告訴使用者並讓他選。

### 5.3 失敗處理

任何 transition 失敗 (permission, validation, network) 都不要重試自動。
告訴使用者:

```
無法將 {KEY} 轉為 {target_status}: {error_message}

請手動在瀏覽器中操作:
{url}
```

## 6. 安全與審計

- token 永遠用環境變數,不寫檔案
- 不在 auto-memory 中存 token (只存 site hostname 這類非機密)
- 每次 Jira 操作都在使用者確認後才執行
- transition 後在 Phase 7.7 完成報告中如實記錄: 哪個 transition / 哪個 resolution
